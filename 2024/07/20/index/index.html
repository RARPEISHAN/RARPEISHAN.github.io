深入设计模型

2024-07-20
OOP
1.2k words

这用来记录我学习深入设计模式记得笔记，便于后续回顾。

在学习前需要基本 UML 语言。

OOP

Object-Oriented Programming is a paradigm：利用抽象、封装、继承以及多态等概念来建立现实实体模型以及之间的交互。

OOD 能够提供的特性：Modularity、Reusability、Scalability、Maintainability、Clear Mapping to Real-World Problem 和 Flexibility and Extensibility

模式

  模式并不是一段特定的代码，而是解决特定问题的一般性概念。要区分模式与算法的区别。两者都是对特定问题的解决方案，但算法提供更明确的步骤，而模式偏向于提供想法。

  1. OOP 主要设计模式
    • Creational Patterns
      • Singleton Pattern
      • Factory Pattern
        • Abstract Factory Pattern
        • Factory Method Pattern
      • Prototype Patten
      • Builder Pattern
    • Structural Patterns
    • Behavioral Patterns

OOD 设计原则

OOD 设计原则： Encapsulation、Abstraction、Inheritance、Polymorphism、Composition 以及 SOLID principle：例如 SRP、OCP、ISP 和 DIP。

  1. 封装变化的内容：找到程序中的 变化内容 并将其与不变的内容分开，减少变化部分对全局的影响。

    • 方法层面的封装
    • 类层面的封装

  2. 面向接口进行开发，而不是面向实现；依赖于抽象类型，而不是 具体类 (concrete classes)

    • 确定一个对象对另一个对象的确切需求：它需要执行哪些方法
    • 在一个新的接口或抽象类中描述这些方法
    • 让被依赖的类实现该接口
    • 让有需求的类依赖于这个接口，而不依赖于具体的类

  3. 组合优于继承，这里列出了几个继承出现的问题

  • 抽象类或非抽象类派生的子类（非抽象类子类），则子类必须实现全部的接口或者抽象方法。抽象类继承抽象类可以部分实现。
  • override 确保新的行为与其基类的版本兼容（这里我觉得意思是函数名需要相同参数个数需要相同等）
  • 继承打破了超类的封装。导致基类实现细节暴露
  • 子类与超类紧密耦合。修改基类同样会影响子类重新编译。
  1. SOLID 原则
  • Single Responsibility Principle
  • Opened/Closed Principle: 对于扩展，类应该是“开放”的；对于修改，类则应该是“封闭”的。
    这里有一点，当类存在缺陷等问题，子类不应该对其父类的问题负责。这个是错误而非修改，不是该原则所应用的类型。
  • Liskov Substitution Principle
